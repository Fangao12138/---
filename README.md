# 区块链版权管理与引用溯源系统

一个面向数字作品的版权登记与流转系统，基于 `Flask + SQLAlchemy + SQLite` 实现业务存储，使用轻量级区块链结构对关键事件（登记、转让、引用）进行上链固化，提供可验证、可追踪的版权管理能力。

## 1. 项目目标

本项目用于解决数字内容场景中的三个核心问题：

- 版权登记后如何证明“作品内容未被篡改”
- 版权流转后如何证明“所有权变更链路真实可信”
- 作品之间存在引用关系时如何进行“上下游溯源”

系统通过“数据库保存结构化业务数据 + 区块链保存关键事件哈希证明”的双层机制，兼顾查询效率与可验证性。

## 2. 核心功能

### 2.1 用户与权限

- 用户注册、登录、退出
- 基于 `Flask-Login` 的会话鉴权
- 上传、转让等敏感操作要求登录后执行

### 2.2 版权登记

- 上传作品文件并计算 `SHA-256` 内容哈希
- 写入版权记录（标题、描述、作者、时间戳、状态）
- 将版权交易信息打包上链，生成区块哈希
- 在详情页查看内容哈希与区块哈希

### 2.3 版权转让

- 版权拥有者发起转让申请
- 受让方确认或拒绝
- 确认后更新作品所有者，并将转让事件上链
- 支持在“所有权验证”页面查看数据库记录与链上记录

### 2.4 引用溯源（新增）

- 上传作品时可填写引用作品 ID（逗号分隔）
- 系统建立 `source_id -> target_id` 引用关系
- 每条引用关系生成 `evidence_hash` 并上链
- 支持溯源接口查询：
  - 上游：当前作品引用了谁
  - 下游：谁引用了当前作品

### 2.5 区块链浏览与验证

- 浏览区块列表、区块详情、交易数量
- 验证链式结构完整性（`previous_hash` 串联）
- 通过区块哈希反查关键业务事件

## 3. 关键原理与方法

### 3.1 文件指纹方法

- 采用 `SHA-256` 对上传文件字节流计算摘要
- 生成固定长度内容指纹 `content_hash`
- 只要文件内容变化，哈希必然变化，从而实现完整性校验

### 3.2 轻量区块链机制

区块包含以下关键字段：

- `index`：区块高度
- `transactions`：交易列表（版权登记/转让/引用）
- `timestamp`：出块时间
- `previous_hash`：前一区块哈希
- `nonce`：工作量证明计数器
- `hash`：当前区块哈希

挖矿流程：

1. 将待处理交易打包
2. 调整 `nonce` 反复计算区块哈希
3. 直到哈希前缀满足难度条件（如前 N 位为 `0`）
4. 区块入链并清空待处理交易池

该机制可提高篡改成本：篡改任一历史区块会破坏后续链条。

### 3.3 数据库存储与链上存证协同

- 数据库存储业务“全量字段”，便于检索与页面渲染
- 链上存储关键交易快照与哈希证明，提供可信背书
- 页面展示时可同时给出业务记录与链上记录，提升可解释性

### 3.4 引用溯源图遍历

溯源接口通过两次图遍历实现：

- 上游遍历：按 `source_id` 递归查找被引用节点
- 下游遍历：按 `target_id` 递归查找引用当前作品的节点

返回数据包含：

- `nodes`：作品节点集合
- `edges`：引用边集合
- `upstream/downstream`：方向化溯源结果
- 每条关系关联 `block_hash`，可追溯到上链记录

## 4. 主要数据模型

### 4.1 Copyright（版权）

- `id`：版权 ID
- `title`：标题
- `description`：描述
- `content_hash`：内容哈希
- `timestamp`：登记时间
- `user_id`：当前所有者
- `block_hash`：登记区块哈希
- `status`：状态（如 `pending/confirmed`）

### 4.2 CopyrightContract（转让合约）

- `copyright_id`：关联版权
- `owner_id / transferee_id`：转让双方
- `status`：转让状态
- `created_at`：创建时间

### 4.3 CopyrightReference（引用关系）

- `source_id`：引用方作品 ID
- `target_id`：被引用作品 ID
- `relation_type`：关系类型（默认 `quote`）
- `evidence_hash`：关系证据哈希
- `block_hash`：该关系的上链区块
- `created_at`：创建时间

## 5. 业务流程说明

### 5.1 版权登记流程

1. 用户上传文件与元数据
2. 系统保存文件并计算 `content_hash`
3. 数据库写入版权记录
4. 版权交易加入待打包池并挖矿
5. 回填 `block_hash`，状态置为已确认

### 5.2 转让流程

1. 所有者发起转让请求
2. 受让方在待处理列表确认/拒绝
3. 确认后更新版权归属
4. 转让交易上链并生成区块证明

### 5.3 引用溯源流程

1. 上传时提交 `reference_ids`
2. 建立引用关系并生成 `evidence_hash`
3. 引用交易上链并回填关系 `block_hash`
4. 通过 `/api/trace/<id>` 拉取上下游链路

## 6. 主要接口

- `GET /`：首页版权列表
- `GET|POST /upload`：作品上传与登记
- `GET /copyright/<id>`：版权详情
- `GET /search`：版权搜索
- `GET /api/verify/<content_hash>`：按内容哈希校验版权
- `GET /api/trace/<copyright_id>`：引用溯源（上下游）
- `GET /blockchain/explorer`：区块浏览器
- `GET /contract/verify/<copyright_id>`：所有权验证

## 7. 项目结构

```text
app/
  models/                 # 数据模型（版权/区块/合约/引用）
  routes/                 # 路由与业务逻辑
  templates/              # Jinja2 页面模板
  static/                 # CSS/JS/上传文件
  blockchain_copyright.db # SQLite 数据库
config.py                 # 配置
run.py                    # 启动入口
requirements.txt          # 依赖
```

## 8. 环境与运行

### 8.1 环境要求

- Python 3.9+
- Windows / Linux / macOS

### 8.2 安装依赖

```bash
pip install -r requirements.txt
```

### 8.3 启动服务

```bash
python run.py
```

启动后访问：`http://localhost:5000`

## 9. 安全与边界说明

- 当前实现用于教学/课程/原型验证，非生产级联盟链
- 默认 SQLite 单机存储，未做多节点共识网络
- 上传目录建议结合对象存储、鉴权下载与病毒扫描
- 建议在生产场景接入审计日志、权限细粒度控制与备份策略

## 10. 后续可扩展方向
